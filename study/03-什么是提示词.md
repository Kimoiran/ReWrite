# 第三课：什么是提示词（Prompt）？

## 3.1 提示词 = 给 AI 的说明书

想象一下，你招了一个新员工。他非常聪明、读过万卷书——但他完全不了解你的项目。

你需要给他写一份说明书：

```
欢迎加入 ReWrite 团队。你的身份是写作助手。

你的工作方式是调用工具，而不是打字描述。

可用的工具有：
  - get_characters：读取角色数据
  - update_character：修改角色字段
  - search_chapters：搜索正文内容
  （共 30 个）

工作规则：
  1. 用户提到内容时，先调 get 工具再回答
  2. 用户说"改"就调 update，说"删"就调 delete
  3. 不要输出「我来检查」——直接调工具
```

这份说明书就是**提示词（Prompt）**，也叫 **System Prompt（系统提示词）**。

**在 ReWrite 中，提示词写在 `prompt_templates.py` 文件的 `DEFAULT_SYSTEM_PROMPT` 变量里。**

## 3.2 提示词长什么样

打开 `src/editor/modules/ai_assistant/prompt_templates.py`，你会看到：

```python
SKILL_MAP = """
工具列表（共 30 个）：

📖 章节
  - get_chapters / read_chapter / create_chapter / ...
👤 人物卡
  - get_character_groups / get_characters / ...
📋 大纲
  - get_outline / update_outline_entry / delete_outline_entry
📅 时间线
  - create_timeline_event / get_timeline / ...
🌍 世界观
  - get_worldview / create_worldview_entry / ...
🗺️ 地图
  - get_map / create_map_node / update_map_node / ...
🔍 搜索
  - search_chapters（语义搜索章节正文，找相关段落）

响应规则（每次回复前必须检查）：
1. 用户提到作品中任何内容时 -> 先调对应的 get 工具再回答
2. 用户说要"写""改""创建""删除" -> 调对应的工具执行
...
"""

DEFAULT_SYSTEM_PROMPT = """你是一个写作助手。你的工作方式不是打字，是调用工具。

""" + SKILL_MAP
```

这个提示词会在**每次对话时都发给 AI**。AI 基于这个提示词来理解自己的身份和工作方式。

## 3.3 提示词是怎么传给 AI 的

提示词不是直接写在代码里的字符串——它会在运行时被组装。

### 第 1 步：获取模板

```python
from prompt_templates import DEFAULT_SYSTEM_PROMPT

system = DEFAULT_SYSTEM_PROMPT
# system = "你是一个写作助手。你的工作方式不是打字，是调用工具。..."
```

### 第 2 步：拼接上下文

```python
# 从 contexts.py 获取当前作品的数据概览
context = collect_context(scope="characters,outline,map")

# 拼接到提示词后面
system = system + "\n\n## 当前作品上下文\n" + context
```

拼完后的提示词长这样：

```
你是一个写作助手。你的工作方式不是打字，是调用工具。

工具列表（共 30 个）：
...

响应规则：
...

## 当前作品上下文
<<<人物设定卡概览>>>
📁 水神
  👤 克诺（十八岁·水神代行者）
📁 智慧
  👤 塞拉丝（一千岁·智慧代行者）
...

<<<地图>>>
📍 克里斯兰 (country)
    📍 特因城 (city)
```

### 第 3 步：加上工具定义

除了文字描述的提示词，API 调用还会附带**结构化的工具定义**：

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_characters",
            "description": "获取人物设定卡",
            "parameters": {  # ← JSON Schema 格式
                "type": "object",
                "properties": {
                    "group": {"type": "string"}
                }
            }
        }
    },
    # ... 所有 30 个工具的定义
]
```

### 第 4 步：发给 API

最终发给 DeepSeek/Claude 的请求体是：

```json
{
  "model": "deepseek-chat",
  "messages": [
    {
      "role": "system",
      "content": "你是一个写作助手。你的工作方式不是打字，是调用工具。\n\n工具列表...\n\n## 当前作品上下文\n..."
    },
    {
      "role": "user",
      "content": "把克诺的年龄改成二十岁"
    }
  ],
  "tools": [...]  // ← 30 个工具的结构化定义
}
```

## 3.4 AI 看到提示词后发生了什么

当 AI 收到上面的请求，它内部的处理逻辑大致是：

```
1. 读取 system prompt → 理解自己是写作助手
2. 读取工具列表 → 知道有 30 个工具可用
3. 读取规则 → 「用户说改就调 update」
4. 读取上下文 → 看到人物卡概览
5. 理解用户消息 → 「把克诺的年龄改成二十岁」
6. 决策：
   - 用户说"改" → 规则 2 要求调工具
   - 需要改角色 → 调 update_character
   - 参数：name=克诺, field=age, value=二十岁
7. 输出工具调用
```

这个过程**不需要任何代码**——完全是 AI 模型根据提示词自己推理出来的。

## 3.5 上下文从哪里来

上下文不是提示词里写死的，是**实时收集**的。在 `contexts.py` 中：

```python
def collect_context(scope, character_module, map_module, ...):
    parts = []
    
    # 如果用户勾选了「人物设定卡」芯片
    if "characters" in scope and character_module:
        text = _characters_text(character_module.nodes)
        parts.append(f"<<<人物设定卡概览>>>\n{text}")
    
    # 如果用户勾选了「地图」芯片
    if "map" in scope and map_module:
        text = _map_text(map_module.nodes, map_module.routes)
        parts.append(f"<<<地图>>>\n{text}")
    
    return "\n\n".join(parts)
```

用户通过 AI 面板上的芯片开关控制 AI 能读到什么：

```
┌──────────────────────────────────────┐
│ AI 可以读取：                          │
│ [当前章节] [人物设定卡] [大纲]          │
│ [世界观] [🗺️ 地图] [时间线]           │
│                                       │
│ [写作助手] [深度分析] [灵感发散]        │
└──────────────────────────────────────┘
```

每个芯片对应一个模块。选中 = 该模块的数据会注入上下文。

## 3.6 提示词的优化过程

在 ReWrite 早期版本中，提示词有 **140 行**。内容非常详细：

```
旧版本（140 行）包含：
- 作品架构说明（章节是什么、人物卡有什么字段）
- 人物卡展示格式（表头、示例行）
- 分析角度（情节逻辑、人物塑造）
- 核心原则 7 条
- 工作流程 5 步
- 人物卡读取原则
```

结果 AI 学会了什么？

**AI 学会了"用文字描述作品"**——因为它被告知了太多关于作品结构的信息。它以为自己的工作是「分析」「描述」「建议」。

新版本（29 行）只保留：

```
新版本（29 行）：
- 一句话身份：「你的工作方式不是打字，是调用工具」
- 工具列表
- 8 条规则
```

AI 现在**只学会了"调工具"**——因为提示词里只有工具和规则，没有多余的写作指导。

### 类比理解

旧提示词像这样教员工：

```
你是我们公司的文案。公司主要做软件开发，我们的产品有...
我们的客户主要是...我们使用的技术栈是...
你的分析角度应该包括：技术可行性、市场需求、用户体验...
```

新提示词像这样：

```
你是程序员。你有 Git、Docker、VS Code。
用户说"改"就改，说"查"就查。
```

**越精简，AI 越聚焦。** 多余的信息会让 AI 分心。

## 3.7 常见的提示词问题

| 现象 | 原因 | 解决 |
|------|------|------|
| AI 只打字不调工具 | 提示词中「分析」「建议」等词太多 | 精简为「执行」「调工具」 |
| AI 调错函数名 | 函数名太长或格式混乱 | 分开展示、加精确规则 |
| AI 不查数据直接回答 | 没告诉 AI「先 get 再回答」 | 加规则 1 |
| AI 输出「我来检查」 | 提示词中用了这类词 | 加规则「禁止输出这类描述」 |

## 3.8 小结

- **提示词**是给 AI 的说明书，定义身份 + 工具 + 规则
- 提示词 + **实时上下文** + **工具定义** 一起发给 API
- 用户通过**芯片开关**控制 AI 能读到什么上下文
- **精简的提示词**比冗长的更有效
- 提示词改在 `prompt_templates.py` 中，改完重启生效
